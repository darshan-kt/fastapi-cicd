# DockerHub Integration with GitHub Actions CI Pipeline

# Overview

This guide demonstrates how to integrate DockerHub into an existing CI pipeline built with:

* FastAPI
* Docker
* GitHub
* GitHub Actions
* DockerHub

The objective is to automatically build and publish Docker images whenever code is pushed to GitHub.

---

# CI Pipeline Architecture

```text
Developer
   |
git push
   |
   v
GitHub Repository
   |
   v
GitHub Actions
+---------------------------+
| Checkout Code             |

| Lint (Ruff)               |
| Run Tests (Pytest)        |
| Coverage                  |

| Build Docker Image        |
| Login to DockerHub        |

+---------------------------+
```

# Why DockerHub?

DockerHub acts as a centralized container registry.

Benefits:

* Store Docker images
* Version container releases
* Share images across environments
* Deploy consistently

# Step 1: Create DockerHub Access Token

Login to DockerHub.

Navigate to:

```text
Account Settings → Personal Access Tokens
```

Create a token:

```text
Name: github-actions
Permissions: Read, Write, Delete
```

Copy the generated token.

**Important:** You will not be able to view it again.

---

# Step 2: Add GitHub Repository Secrets

Open GitHub repository:

```text
Settings → Secrets and variables → Actions → New repository secret
```

Add the following repository secrets.

## Secret 1

Name:

```text
DOCKERHUB_USERNAME
```

Value:

```text
your_dockerhub_username
```

---

Note: Select again New repository

## Secret 2

Name:

```text
DOCKERHUB_TOKEN
```

Value:

```text
your_dockerhub_access_token
```

After adding, you should see:

```text
✓ DOCKERHUB_USERNAME
✓ DOCKERHUB_TOKEN
```

# Step 3: Verify Dockerfile

Example:

```dockerfile
FROM python:3.12-slim

WORKDIR /app

COPY requirements.txt .

RUN pip install --no-cache-dir -r requirements.txt

COPY . .

EXPOSE 8000

CMD ["uvicorn", "app.main:app", "--host", "0.0.0.0", "--port", "8000"]
```

Build locally:

```bash
docker build -t fastapi-cicd .
```

Run:

```bash
docker run -p 8000:8000 fastapi-cicd
```

Test:

```bash
curl localhost:8000
```

Expected:

```json
{
  "status": "healthy",
  "service": "fastapi-cicd"
}
```

# Step 4: Configure GitHub Actions

Update:

```text
.github/workflows/ci.yml
```

Example:

```yaml
name: CI

on:
  push:
    branches:
      - main

  pull_request:

jobs:
  ci:
    runs-on: ubuntu-latest

    env:
      PYTHONPATH: .

    steps:
      - uses: actions/checkout@v4

      - uses: actions/setup-python@v5
        with:
          python-version: "3.12"

      - name: Install Dependencies
        run: |
          pip install -r requirements.txt
          pip install -r requirements-dev.txt

      - name: Lint
        run: ruff check .

      - name: Test
        run: pytest -v

      - name: Coverage
        run: |
          coverage run -m pytest
          coverage report

      - name: Security Scan
        run: bandit -r app

      - name: Login to DockerHub
        if: github.event_name == 'push'
        uses: docker/login-action@v3
        with:
          username: ${{ secrets.DOCKERHUB_USERNAME }}
          password: ${{ secrets.DOCKERHUB_TOKEN }}

      - name: Build and Push Image
        if: github.event_name == 'push'
        uses: docker/build-push-action@v6
        with:
          context: .
          push: true
          tags: |
            ${{ secrets.DOCKERHUB_USERNAME }}/fastapi-cicd:latest
            ${{ secrets.DOCKERHUB_USERNAME }}/fastapi-cicd:${{ github.sha }}
```

# Why Tag with Git SHA?

Example:

```text
myuser/fastapi-cicd:a7c91f2
```

Benefits:

* Immutable releases
* Easy rollback
* Full traceability
* Versioned deployments

Avoid deploying:

```text
latest
```

to production.

Prefer:

```text
a7c91f2
```

# Common Failures

## Docker Login Failure

Error:

```text
unauthorized: incorrect username or password
```

Check:

* Secret names
* Token validity
* DockerHub username

---

## Push Denied

Error:

```text
requested access to the resource is denied
```

Possible causes:

* Repository does not exist
* Wrong username
* Insufficient permissions

---

## Build Failure

Debug locally:

```bash
docker build -t test .
docker run -p 8000:8000 test
```

# Best Practices

✅ Use immutable image tags
✅ Push images only on `main` branch
✅ Never store secrets in Git
✅ Build once, deploy many times
✅ Test before Docker build
✅ Use access tokens instead of passwords

# Interview Questions & Answers

## 1. Why use DockerHub instead of building during deployment?

**Answer:**

Building once in CI ensures the same image is deployed everywhere.

```text
Build once → Deploy many times
```

This improves consistency and reliability.

---

## 2. Why should Docker images be immutable?

**Answer:**

Immutable images never change after creation.

Benefits:

* Easier rollback
* Predictable deployments
* Better traceability

---

## 3. Why tag Docker images with Git SHA?

**Answer:**

Git SHA uniquely identifies the source code version used to build the image.

Example:

```text
myapp:a7c91f2
```

This enables:

* Rollbacks
* Auditing
* Reproducibility

---

## 4. Why should pull requests not push images?

**Answer:**

PRs may contain incomplete or unapproved code.

Publishing images only after merging to `main` prevents unnecessary or invalid artifacts.

---

## 5. What happens if DockerHub is unavailable?

**Answer:**

The CI pipeline may fail to publish images.

Mitigations:

* Retry mechanisms
* Alternative registries
* Cached images
* Separate build and publish stages

---

## 6. Why use GitHub Secrets?

**Answer:**

Secrets securely store sensitive values such as:

* DockerHub tokens
* Cloud credentials
* API keys

Never commit secrets into source control.

---

## 7. How do you debug a failing Docker push?

**Answer:**

Check:

```bash
docker login
docker build
docker push
```

Verify:

* Credentials
* Repository name
* Permissions

Read GitHub Actions logs carefully.
